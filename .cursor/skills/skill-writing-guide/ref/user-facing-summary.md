# 建立／優化完成後：給使用者的簡報

套用本 guide **完成**新建或優化某 skill 後，**同一則回覆末尾**須有一段簡短說明，讓使用者知道「這顆算類」與「這次動了哪些寫作法」。不需長篇；重點是可掃描。

---

## 必含兩塊

### 1）主九類（擇一）

從 `nine-techniques.md` 九類表選一個**主用途**；名稱用表內英文，後面可加括號繁中輔助。

- 若內容**明顯橫跨多類**：仍標主類，並加一句「建議未來拆成 … / …（簡述）」；若這次已拆，寫「已拆出／合併至 …」。

### 2）本次採用的撰寫／優化手段（對照九技巧）

從下面對照表勾 **本次實際有做到** 的項目（用編號即可，必要時一行註解）。**未動到的技巧不必列出**。

| 編號 | 短標籤（可當簡報用語） |
|:---:|:--|
| 1 | 只寫專案特有、推離預設 |
| 2 | Gotchas（新增／補條） |
| 3 | 漸進式揭露（hub + `ref/`／`assets/`） |
| 4 | 目標+約束，不寫死逐步驟 |
| 5 | `description` 觸發導向 |
| 6 | `config.json` 首次設定（有做才列） |
| 7 | 穩定目錄記憶（有做才列） |
| 8 | 可複製片段／範本／`assets` |
| 9 | 按需 hook（有做才列） |

**優化**時可多加一行 **相對前一版**：例如「補 description 觸發」「`SKILL.md` 瘦身改 hub」「新增 `ref/xxx.md`」。

---

## 建議格式（可直接貼給使用者）

```markdown
**Skill 分類**：Scaffolding & Templates（腳手架與範本）

**本次採用**：③ hub + `ref/` · ⑤ `description` 觸發句 · ② 補 2 條 Gotchas  
**相對前一版**（若有）：將長篇 API 拆到 `ref/api.md`
```

單顆 skill 用一小段即可；**批次審查**時可在摘要表每列加「主類 | 本次建議手段編號」欄，見 `review-workflow.md`。

---

## 避開

- 不要只寫「已依指南優化」卻不標類別與編號（使用者無法對照累積經驗）。
- 不要把九技巧全文複製一遍；只列**有做到**的編號／短標籤。
- `description` 勿加 `Trigger on:` 前綴（技巧 5）。
