# 審查其他 Skill 的工作流

給「要檢查一整批 `**/skills/**`」時用：不必機械式逐檔念完，用**兩階段**較省 token、也比較不容易漏大項。

---

## 建議：兩階段，不要只「一個一個慢慢讀全文」

| 階段 | 做什麼 | 為什麼 |
|------|--------|--------|
| **A. 批次掃描** | 對每個 skill 只讀 `SKILL.md` 前 40 行內：`description`、`## Gotchas` 有無、hub 是否指到 `ref/` | 快速抓「結構性違規」（觸發詞、缺 Gotchas、單檔過長） |
| **B. 按需深讀** | 僅對 A 有問題的 skill、或「高頻 / 高風險」skill 再讀完 `ref/`、`scripts/` | 避免每個 skill 都全文載入 |

**結論**：可以「一個一個 skill 當作**清單上的條目**」跑完，但**每條目先做輕量檢查**；不要預設每條都深讀全部子檔。若使用者要求「每個都要完整審到 ref」，再切到全量深讀。

---

## A 階段檢查項（每個 skill 約 1 次讀檔）

- `name:` 與資料夾名一致（kebab-case）
- `description` 是否像**觸發條件**（何時載入、關鍵字／情境），而非功能行銷句；是否誤把**執行約束**寫進 `description`（應改到 **`constraints:`** 或正文／`ref/`）
- 是否有 **`Gotchas`** 區塊（可空，但區塊名要存在）
- `SKILL.md` 是否過長（遠超 ~50 行卻沒拆 `ref/`）→ 建議拆 hub
- 九類是否**明顯**錯置（一個 skill 混很多類）→ 建議拆分（見本目錄 `nine-techniques.md` 九類表）

---

## B 階段何時開深讀

- A 階段任一項失敗
- 該 skill 常被觸發、或牽涉刪檔 / 部署 / 對外 API
- 使用者明定「要審腳本與 ref」

深讀時對照：`nine-techniques.md` 九技巧 + hub 內「優化其他 skill 時的採用規則」。

---

## 產出建議（給使用者）

- **摘要表**：每 skill 一行 — 狀態（OK / 需改）、**主九類**（見 `nine-techniques.md`）、主因、建議動作（補 description、拆 ref、加 Gotcha…）；若本次已代為優化，加欄 **採用手段**（九技巧編號 1～9，見 `user-facing-summary.md`）
- **詳細**：只列「需改」的 skill，避免把 20 個 OK 各寫一段
- **單顆新建／優化完稿**：須依 `user-facing-summary.md` 給使用者一段「主類 + 本次技巧」簡報（與 hub `SKILL.md` 相同要求）

---

## 與「不要寫死 Step」的關係

審查流程本身允許**檢查清單**（可重複、可掃描）；被審的 skill 內文仍應避免把業務流程寫成死板 Step 1/2/3（見 `nine-techniques.md` 技巧 4）。
